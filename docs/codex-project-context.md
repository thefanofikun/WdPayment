# Codex 协作上下文

## 1. 项目长期目标

这个仓库的长期目标不是继续做留言板，而是逐步演进为一套：

`ToB 法币支付平台`

约束如下：

- 后端：`Java`
- 前端：`Vue`
- 终端形态：网页
- 当前阶段：先不对外开放 API
- 核心用户：商户、运营、财务、风控、审核
- 建设方向：企业级、可上线、可持续扩展

## 2. 和 Codex 协作时的默认原则

每次新对话开始时，优先以这个目标理解需求：

1. 优先服务支付平台长期建设
2. 优先沉淀能复用的系统设计和工程结构
3. 优先建设核心中台，而不是只做演示页面
4. 所有设计都尽量考虑可审计、可追溯、可运维
5. 在 0 到 1 阶段优先选择 `模块化单体`

## 3. 当前推荐优先级

如果没有特别说明，默认按下面顺序推进：

1. 认证与权限
2. 商户管理
3. 支付订单
4. 账户系统
5. 账务总账
6. 商户门户
7. 运营后台
8. 清结算
9. 对账
10. 风控

## 4. 每次继续协作的建议做法

为了让我在新的对话里尽快进入状态，建议你在开头直接说一句：

`请先阅读 docs/codex-project-context.md 和 docs/payment-platform-blueprint.md，再继续当前任务。`

这样我会先读取这两份文档，再基于同一目标继续工作。

## 5. 这些文档解决什么问题

这些文档不能让模型永久记忆你的目标，但可以把目标保存在仓库里。

好处是：

- 目标不会随着聊天结束而丢失
- 后续每次都能从仓库里恢复上下文
- 系统边界、优先级和建设路线不需要反复重讲

## 6. 当前阶段最值得继续做的事情

下一批高价值工作建议是：

1. 先建设多通道网关系统，统一上游接口，对接多个下游通道
2. 当前优先覆盖 onboarding、VA、beneficiary、payout、webhook 五类能力
3. 前端先做独立 Vue 控制台，便于快速联调和演示
4. 后端先用 Spring Boot 做模块化骨架，后续再接入真实通道
